# Claude への単発レビュー依頼

Claude Code CLI の `claude -p` を使い、単発のレビュー結果を標準出力で受け取ります。
agmsg、`spawn`、常駐 Claude セッション、ターミナル操作は使いません。

## 前提

- `claude -p` は非対話で応答を出力して終了します。
- レビュー用の材料は Codex 側で集め、Claude には必要最小限のテキストとして渡します。
- Claude にファイル編集をさせません。レビュー結果だけを返させます。
- 既存のローカルファイルや差分を大量に丸投げせず、対象範囲を絞ります。
- 機密値、不要なログ全文、認証情報、個人情報はプロンプトに含めません。

## ワークフロー

### 1. CLI を確認する

`claude` が使えるか確認します。

```bash
command -v claude
claude -p --help
```

`claude` が見つからない場合、または `-p` が使えない場合は、レビューを実行できないことをユーザーに報告します。

### 2. レビュー対象を整理する

必要な範囲だけを確認します。

- 対象: uncommitted diff、staged diff、特定ファイル、設計メモ、PR 相当の変更など。
- 目的: バグ検出、回帰リスク、テスト観点、設計レビュー、セキュリティ確認など。
- 現状: ブランチ名、主な変更点、既に実行したテスト、失敗している検証。
- 制約: 編集禁止、既存パターン優先、広いリファクタ禁止、特定ファイルだけを見る、など。

コードレビューなら、まず `git diff --stat`、`git diff --name-only`、必要に応じて `git diff` または `git diff --staged` を確認します。
差分が大きい場合は、ユーザーの指定やリスクの高いファイルに絞ります。

### 3. Claude に渡すプロンプトを作る

Claude には、レビュー担当としての役割、対象、観点、制約、出力形式、レビュー材料を明示します。

```text
あなたはコードレビュー担当です。以下の変更をレビューしてください。

制約:
- ファイルは編集しないでください。
- レビュー結果だけを返してください。
- 指摘は重大度順に並べてください。
- 可能なら file:line を示してください。
- 根拠が弱い推測は「要確認」と明記してください。
- 問題がなければ「重大な指摘なし」と書いてください。

レビュー観点:
- バグ、回帰、境界条件
- テスト不足
- 既存設計との不整合
- セキュリティまたはデータ破壊リスク

対象プロジェクト: <absolute project path>
現在の状況: <summary and tests>

レビュー材料:
<diff, selected files, or plan>
```

### 4. `claude -p` を実行する

短いプロンプトなら引数で渡します。

```bash
claude -p --no-session-persistence --permission-mode dontAsk --tools "" "<prompt>"
```

長いレビュー材料を渡す場合は、標準入力を使います。

```bash
printf '%s\n' "<prompt>" | claude -p --no-session-persistence --permission-mode dontAsk --tools ""
```

`--tools ""` は Claude 側のツール使用を無効化し、Codex が集めた材料だけでレビューさせるために使います。
Claude に追加調査をさせたい明確な理由がある場合だけ、ユーザーに説明したうえで `--tools` や `--allowedTools` を変更します。

`claude -p` が認証、権限、ネットワーク、予算、または CLI エラーで失敗した場合は、コマンドの要点と失敗理由を報告します。未ログインの場合は `Not logged in` / `Please run /login` のようなエラーになります。失敗時に agmsg や `spawn` へ自動でフォールバックしません。

### 5. 結果を扱う

Claude の指摘はそのまま採用せず、対象コードや差分と照合します。
存在しないファイル、古い差分、再現不能な指摘、ユーザーの制約に反する提案は除外または保留します。

ユーザーへの報告では、Claude の指摘と Codex 側で確認した事実を分けます。
修正まで依頼されている場合は、確認できた指摘だけを通常の編集ワークフローで扱います。
