# 技術的な手順書の作成

日本語の技術手順書は、読者が迷わず作業し、結果を判断し、失敗時に止まれる文書にする。
説明文ではなく、作業の条件、手順、確認、復旧を揃えた実行可能な文書として書く。

## 併用スキル

- `japanese-tech-writing` は必ず適用する。
  一文一行、論理の一方向性、冗長な総括の排除、確認していないことを確認済みのように書かない規範に従う。
- インフラ作成や変更のコマンドを含む場合は、`write-infra-console-expectations` も使う。
  対象は `aws`、`terraform apply`、`terraform plan`、`cloudformation deploy`、`gcloud`、`az`、`kubectl apply`、その他クラウドや Kubernetes リソースを作成・変更する手順である。
- 文書が `.docx`、PDF、スライド、スプレッドシートなどのファイル形式そのものを扱う場合は、対応する文書系スキルも併用する。

## 作成手順

1. 目的と読者を決める。
   読者が何を終えればよいのか、どの権限と前提知識を持つのかを明確にする。
2. 作業範囲を分ける。
   準備、実行、確認、切り戻し、後片付けを混ぜずに章立てする。
3. 前提条件を書く。
   必要な権限、環境、バージョン、接続先、作業ディレクトリ、未完了なら止める条件を書く。
4. 手順を書く。
   ひとつの番号にひとつの操作を置き、コマンドと編集箇所をコードブロックで示す。
5. 期待結果を書く。
   コマンド出力、画面表示、ファイル差分、ログ、メトリクスなど、読者が成功を判断できる値を示す。
6. 失敗時の扱いを書く。
   中断条件、よくあるエラー、再実行可否、切り戻し手順、問い合わせに必要な情報を示す。
7. 文章を整える。
   `japanese-tech-writing` の規範で、冗長な説明、曖昧な主語、確認していない断定を削る。

## 不足情報の扱い

手順書に必要な情報が足りないときは、作業を進める前に不足を分類する。

- **作業を止める不足**：対象環境、作業対象、破壊的操作の影響範囲、不可逆操作の境界、承認条件、切り戻し元、秘密情報の扱いが不明な場合。
  最終版の手順書として納品する前に、ユーザーへ確認するか、本文の冒頭に「要確認」として明示する。
- **プレースホルダーで進められる不足**：実リソース名、アカウント名、VPC 名、保持期間、暗号化方式、社内 CLI 名など、文書の型は決まるが実値だけが未提供の情報。
  `<production-db-id>` のように置換対象だと分かる形で書き、実値を捏造しない。
- **条件付きで書く不足**：リソース間の関連、ログ出力設定、通知設定、監視連携など、存在する可能性はあるが入力から断定できない情報。
  「設定している場合は」「Terraform 定義で指定している場合は」のように条件を付ける。

不足情報を補うために、実在するコマンド、設定値、画面名、秘密情報を作らない。
未確認の値は「想定」「確認する」「指定した値」の形で書き、確認済みの事実として扱わない。

プレースホルダーや条件付き項目が残る場合は、手順書内に「要確認事項」または「置換値」の節を置く。
その節には、作業前に埋める値、確認元、未確認のまま実行してはいけない条件を書く。
不足情報を安全に処理して本文へ反映した場合は、その情報を未解決の不明点として残さない。

## 標準構成

既存文書の型がなければ、次の順に構成する。

````markdown
# 手順名

## 目的

この手順で達成する状態を書く。

## 対象読者

必要な権限、担当範囲、前提知識を書く。

## 前提条件

- 必要なツールとバージョン
- 作業対象の環境
- 必要な権限
- 作業前に確認する状態

## 要確認事項

- `<placeholder>`：確認元と埋める値
- 条件付きの設定：有効な場合だけ実施する確認
- 未確認のまま実行してはいけない条件

## 手順

1. 操作を書く。

   ```sh
   command
   ```

   期待される結果を書く。

## 確認

成功を判断する確認項目を書く。

## 切り戻し

失敗時または中止時に戻す手順を書く。

## 注意点

破壊的操作、課金影響、停止時間、権限、個人情報、秘密情報に関する注意を書く。
````

## 手順の書き方

- コマンドの前に、何のために実行するかを一文で書く。
- コマンドの後に、成功時に何を見るかを書く。
- 実行ディレクトリ、環境変数、プロファイル、リージョン、namespace、対象ブランチなど、結果を変える条件は省かない。
- 破壊的な操作は、実行前の確認条件と中止条件を直前に置く。
- `sudo`、本番環境、データ削除、権限変更、公開範囲変更、課金増加を伴う手順は、影響範囲を明示する。
- 長い背景説明は手順の途中に入れない。
  判断に必要な背景だけを「目的」または「注意点」に置く。
- 既存の手順書を改稿する場合は、見出し階層、語尾、番号付け、注意書きの形式を合わせる。

## 確認の書き方

確認は、作業者が成功と失敗を区別できる粒度で書く。

- コマンド確認：終了コード、標準出力、ファイル差分、API レスポンス、ログ行などを書く。
- 画面確認：画面名、対象識別子、状態、主要設定、反映までの待ち時間を書く。
- 動作確認：HTTP ステータス、画面表示、ジョブ状態、メトリクス、通知到達などを書く。
- データ確認：件数、代表レコード、更新時刻、移行前後の差分などを書く。

AWS CLI や Terraform などでクラウドリソースを作成する場合は、`write-infra-console-expectations` の基準で Web コンソール確認を書く。

## 切り戻しの書き方

切り戻しは、単なる「元に戻す」ではなく、実行可能な作業として書く。

- どの時点までなら切り戻せるかを書く。
- 再実行してよいコマンドと、再実行してはいけないコマンドを分ける。
- バックアップ、スナップショット、旧設定、旧バージョン、直前のコミットを使う場合は、参照先を明示する。
- 切り戻し後の確認項目を書く。
- 切り戻せない操作は、実行前の承認条件や停止条件を強く書く。

## 最終点検

書き終えたら、次を確認する。

- 読者、対象環境、作業範囲が冒頭で分かる。
- 各手順に、操作、期待結果、失敗時の判断材料がある。
- コマンドや設定値が、プレースホルダーなのか実値なのか区別できる。
- 確認済みでない内容を、確認済みとして断定していない。
- 秘密情報、個人情報、アクセスキー、トークンを本文に含めていない。
- `japanese-tech-writing` の規範に反する冗長な総括、空虚な強調、曖昧な比喩がない。
